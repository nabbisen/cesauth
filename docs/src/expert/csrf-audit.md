# CSRF audit (v0.24.0)

Survey of every HTTP route in `cesauth-worker` that accepts state-
changing requests, with a determination of which CSRF defense
applies and verification that the implementation matches the
documented model in [`csrf.md`](./csrf.md).

This audit is a one-time deliverable for v0.24.0. The result of
the audit is captured below; the audit methodology is documented so
that future contributors can re-run it for new routes.

## Summary

**No critical gaps found.** The pre-existing CSRF infrastructure
(token-based for browser-cookie-authenticated form POSTs;
Origin/Referer check for state-clearing POSTs; bypass for
JSON-only and bearer-authenticated paths) is correctly applied to
every state-changing route. One small gap was filled in v0.24.0:
the `/magic-link/verify` form path now validates the CSRF token
in the same way `/magic-link/request` does (defense-in-depth;
the route was already practically unforgeable because both
`handle` and `code` are server-issued secrets, but the documented
model in `csrf.md` claimed protection that wasn't there).

A pre-existing UX gap was discovered: the
`magic_link_sent_page()` template produces a form that's missing
the `handle` and `csrf` hidden fields. This makes the HTML form
path unusable in browsers (only the JSON API path works); it's
not a security issue (the broken form fails-closed on the empty-
handle check) but it's a documented bug that needs a separate
fix. Tracked as a follow-up in ROADMAP.

## Methodology

1. **Enumerate every route** that accepts POST/PUT/PATCH/DELETE.
2. **For each route**, identify the authentication mechanism it
   uses: cookie-based session, `Authorization: Bearer`, OAuth
   client credentials, or unauthenticated.
3. **Identify the CSRF defense** that applies:
   - **CSRF token** for cookie-authenticated form POSTs from
     browsers.
   - **Origin/Referer check** for state-clearing routes where
     adding a CSRF token would break programmatic clients
     (e.g., `/logout`).
   - **CORS preflight** for JSON-only POSTs (cross-origin
     `application/json` requires preflight, which the deployment
     does not grant).
   - **`Authorization` header** for bearer-authenticated routes
     (browsers cannot auto-send `Authorization` from a
     cross-origin form).
   - **No defense needed** for unauthenticated read-only routes.
4. **Verify** the route's implementation actually applies the
   identified defense.

The route inventory was generated by:

```sh
grep -rn '\.post_async\|\.put_async\|\.patch_async\|\.delete_async' \
  crates/worker/src/lib.rs crates/worker/src/routes/
```

## Route inventory

### Browser-cookie-authenticated form POSTs

These routes accept `application/x-www-form-urlencoded` from
HTML form submissions and authenticate via the
`__Host-cesauth_session` cookie (which is `SameSite=Lax`, so
top-level cross-site form POSTs *would* carry the cookie absent
other defenses). CSRF token defense is **required** here.

| Route | CSRF defense | Implementation site |
|---|---|---|
| `POST /magic-link/request` | CSRF token | `crates/worker/src/routes/magic_link/request.rs` |
| `POST /magic-link/verify`  | CSRF token (added in v0.24.0) | `crates/worker/src/routes/magic_link/verify.rs` |

### State-clearing form POST

| Route | CSRF defense | Implementation site |
|---|---|---|
| `POST /logout` | Origin/Referer check | `crates/worker/src/routes/session.rs` |

`/logout` uses an Origin/Referer check rather than a CSRF token
because the route has no UI form preceding it (operators and
integration tools call it directly via `curl`/`fetch`). The
Origin check accepts requests where `Origin` (or, fallback,
`Referer`) matches the deployment's `rp_origin`. Requires the
caller to send at least one header, which any real browser does
on POST.

`SameSite=Lax` on the session cookie is the primary defense (a
cross-origin POST from an attacker page does not carry the
cookie, so a forged logout is a no-op anyway). The Origin check
is defense-in-depth.

### JSON-only POSTs (CORS-protected)

These routes accept only `application/json`. A cross-origin
`application/json` POST triggers a CORS preflight, which
cesauth does not grant to arbitrary origins. So an attacker
page cannot construct one from the victim's browser.

| Route | Defense |
|---|---|
| `POST /webauthn/register/finish`     | CORS preflight |
| `POST /webauthn/authenticate/finish` | CORS preflight |
| `POST /api/v1/anonymous/promote`     | CORS preflight |

### Bearer-token-authenticated routes

These routes require `Authorization: Bearer <token>` set by an
HTTP client (curl, browser extension, programmatic integration).
A browser cannot auto-send an `Authorization` header from a
cross-origin form submission — the header would have to be set
by JS, and JS is same-origin. So an attacker page cannot forge
authentication for these routes.

| Route family | Auth |
|---|---|
| `POST /admin/console/*`           | Admin bearer |
| `POST /admin/tenancy/*`           | Admin bearer |
| `POST /admin/t/<slug>/*`          | User-as-bearer (per ADR-002) |
| `POST /api/v1/...`                | Admin bearer |

The current implementation reads bearer tokens **only from the
`Authorization` header**; there is no cookie-fallback path.
This is verified in `crates/worker/src/routes/admin/auth.rs`'s
`bearer()` function. If a future release adds a cookie path
(planned at v0.10.0+ per old comments, never implemented), the
audit must be re-run for those routes — a cookie-authenticated
admin form **would** be CSRF-able and would need a token check.

### OAuth machine-to-machine routes

| Route | Auth |
|---|---|
| `POST /oauth/token`     | OAuth client_secret + grant-specific secret |
| `POST /oauth/revoke`    | OAuth client_secret |
| `POST /oauth/introspect` | (planned) OAuth client_secret |

These are not browser-originated. Browsers do not visit
`/oauth/token` directly; the OAuth flow is server-to-server from
the client (Relying Party). An attacker would have to compromise
the client's `client_secret` — which is a different attack
surface, not a CSRF issue.

### Unauthenticated public reads

GET-only and have no state-changing effect:

- `GET /.well-known/openid-configuration`
- `GET /.well-known/jwks.json`
- `GET /authorize` (sets the CSRF cookie for the subsequent POST,
  but doesn't change state itself)
- `GET /login`
- `GET /` (landing page)

No CSRF defense needed.

## Cookies and SameSite audit

| Cookie | SameSite | Verdict |
|---|---|---|
| `__Host-cesauth_session` | `Lax` | Correct — `Strict` would break OAuth callback flows where the redirect from the RP back to cesauth needs the session to follow |
| `__Host-cesauth-csrf`    | `Strict` | Correct — token cookie should never travel cross-site |
| `__Host-cesauth_pending` | `Lax` | Correct — same reasoning as session cookie; pending cookie must survive OAuth callback chain |

All three cookies use the `__Host-` prefix, which forces
`Secure`, `Path=/`, and disallows `Domain` — preventing both
overwrite-from-subdomain attacks and accidental cookie leakage to
non-HTTPS contexts.

## Token-binding analysis

The CSRF token is bound to the `__Host-cesauth-csrf` cookie, not
to a session id. This is correct for the threat model:

- The cookie can only be set by the cesauth server (`__Host-`
  prefix prevents cross-origin `Set-Cookie`). An attacker page
  on `attacker.com` cannot pre-fix the cookie value.
- An attacker page also cannot read the cookie (`HttpOnly`).
- The token in the form field is rendered server-side by the
  same response that sets the cookie. They match by construction.
- If the user has multiple browser tabs/sessions open, all share
  the same `__Host-cesauth-csrf` cookie value. This is
  acceptable: a CSRF token from one tab's form is not a CSRF
  risk for another tab — both belong to the same user.

A "session-bound" CSRF token (one that's tied to the session id
rather than a separate cookie) would offer no additional
protection here because the existing model already prevents
the attacks token-binding aims to mitigate.

## Discovered pre-existing UX bug

The HTML template `magic_link_sent_page()` (in
`crates/ui/src/templates.rs`) renders a form that POSTs to
`/magic-link/verify` but is missing both the `handle` and `csrf`
hidden input fields. The verify handler requires both. So the
form path is currently broken in browsers — only the JSON API
path works.

**Impact**: not a security issue (the broken form fails closed —
the handler returns 400 on the empty-handle check before any
state change). This is a UX bug.

**Fix scope**: out of scope for v0.24.0 (CSRF audit). Tracked as
a separate ROADMAP item. The fix needs:
1. `magic_link_sent_page()` to take the handle and CSRF token as
   parameters.
2. The `/magic-link/request` handler to pass them to the
   template.
3. New tests for the full form-path flow.

## Recommendations for new routes

When adding a new route, follow this decision tree:

1. **Does the route accept state-changing requests?** No → no
   CSRF defense needed.
2. **Is the route's authentication browser-cookie-based?**
   - Yes, and the route accepts forms → CSRF token via
     `crate::csrf::verify`.
   - Yes, but no UI form (state-clearing) → Origin/Referer
     check via `crate::csrf::check_origin_or_referer`.
   - No (Authorization header / OAuth client_secret) → no
     CSRF defense needed; document this in the route's module
     docstring.
3. **Is the route JSON-only?** Document this; verify that
   `application/x-www-form-urlencoded`,
   `application/multipart/form-data`, and `text/plain` are
   all rejected. CORS preflight does the rest.

The pure-function library (`crate::csrf`) provides:

- `csrf::mint()` — generate a token.
- `csrf::set_cookie_header(token)` — build the Set-Cookie value.
- `csrf::extract_from_cookie_header(cookie_header)` — read the
  token from a request's `Cookie` header.
- `csrf::verify(submitted, from_cookie)` — constant-time
  compare.
- `csrf::check_origin_or_referer(origin, referer, expected)` —
  Origin/Referer validation.

## Test coverage

The CSRF library has 16 unit tests in
`crates/worker/src/csrf/tests.rs` covering:

- `mint()` produces 32-byte base64url tokens (24 bytes of
  CSPRNG → 32 chars base64url-no-pad).
- `mint()` calls produce distinct values.
- `extract_from_cookie_header` finds the cookie among other
  cookies (delimiter handling).
- `extract_from_cookie_header` returns None for missing cookie.
- `verify` accepts equal tokens.
- `verify` rejects empty inputs (both submitted and cookie).
- `verify` rejects mismatched tokens in constant time.
- `check_origin_or_referer` accepts same-origin Origin.
- `check_origin_or_referer` rejects cross-origin Origin.
- `check_origin_or_referer` rejects `Origin: null`.
- `check_origin_or_referer` falls back to Referer when Origin
  absent.
- `check_origin_or_referer` rejects Referer prefix attack
  (e.g., `https://cesauth.example.com.attacker.com/...`).
- `check_origin_or_referer` rejects when both headers absent.
- `check_origin_or_referer` rejects empty `expected_origin`.
- `constant_time_eq` rejects different-length inputs.
- `constant_time_eq` accepts identical inputs.

Plus the v0.24.0-added `VerifyBody` / `RequestBody`
deserialization tests in the magic-link route handlers (4 +
2 = 6 tests). These pin the contract that the CSRF field
correctly flows from form/JSON input to the gate.

Total CSRF-related tests: **22**.

## Re-audit cadence

This audit should be re-run:

- **At each major release** (v0.x.0 → v1.0.0, v1.0.0 → v2.0.0).
- **Whenever a new mutating HTTP route is added.** The route's
  PR description should call out which CSRF defense applies and
  why.
- **Whenever the auth model changes** — for example, if a
  cookie-fallback path for admin tokens is added (planned but
  never shipped).
- **Whenever a CSRF-related dependency** (cookie crate, base64
  encoding) is upgraded.
