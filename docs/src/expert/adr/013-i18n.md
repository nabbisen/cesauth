# ADR-013: Internationalization infrastructure (i18n track)

**Status**: **Accepted (v0.36.0)**.

**Context**: cesauth's user-facing UI through v0.35.0 was
language-mixed without negotiation. The end-user TOTP /
Security Center / login / sessions surfaces carried
hardcoded Japanese strings; the admin console carried
hardcoded English; machine-emitted error bodies were
English. There was no `Accept-Language` handling, no locale
negotiation, no message catalog, no fallback chain. The
v0.31.0 flash module carried a TODO at
`crates/worker/src/flash.rs:215` ("Translations are out of
scope for v0.31.0; future i18n would replace this lookup
with one keyed by Accept-Language") — an explicit i18n debt.

ROADMAP listed i18n as a parallel track with four phases
(i18n-1 infrastructure, i18n-2 ja+en MVP, i18n-3 additional
locales on demand, i18n-4 RFC 5646 region/script subtag
edge cases). v0.36.0 ships **i18n-1 plus partial i18n-2** —
the infrastructure plus initial migration of the highest-
visibility surfaces.

## Decision

### `cesauth_core::i18n` is the catalog

The catalog lives in `cesauth-core` (not a separate crate).
Every other crate already depends on `cesauth-core`; adding
a separate `cesauth-i18n` crate would only add a layer of
indirection. The catalog colocates with the domain types
that already define enums for closed sets of user-visible
states (`AuthMethod`, `FlashKey`, `EventKind` etc.).

Three exports:

```rust
pub enum Locale { Ja, En }            // closed set
pub enum MessageKey { /* PascalCase */ }  // closed set
pub fn lookup(MessageKey, Locale) -> &'static str
pub fn parse_accept_language(&str) -> Locale
```

### Closed `Locale` enum, recompile to add

Adding a locale requires a new variant + match arms in
every `lookup` `match Locale` block. The compiler enforces
this via match exhaustiveness — partial coverage is a
build error.

Alternative considered: runtime catalog loading from a
filesystem directory. **Rejected**: would be more attack
surface (filesystem trust, directory-traversal risks if the
loader gets the path resolution wrong) for marginal benefit
(translators can't ship a locale without a redeploy, but
neither can they without test coverage anyway). The closed
enum also lets the call sites use Rust's exhaustiveness
checking when they branch on locale, e.g. for date-format
selection in future iterations.

### Closed `MessageKey` enum

Every variant has every locale's translation, statically
guaranteed. Naming convention: PascalCase named after the
*concept* (e.g., `TotpEnabled`), not the rendered text. So
renaming or rephrasing the text in one locale doesn't
require renaming the key in code.

Adding a key:

1. Add a variant to `MessageKey`.
2. Add a `match locale` arm in `lookup` for the new
   variant — the compiler enforces full coverage of the
   `Locale` axis automatically.
3. Replace the call site's hardcoded literal with
   `lookup(key, locale)`.

A test (`every_message_key_resolves_in_every_locale_to_nonempty`)
pins that no translation slot is silently empty.

### `parse_accept_language` is RFC 7231 §5.3.5 (q-value-aware)

Lenient subset of the spec:

- Splits on `,` for entries; per-entry splits on `;` for
  parameters.
- Recognizes `q=<float>`, default 1.0.
- Strips region subtags (`ja-JP` → `ja`, `en-US` → `en`).
  Region-aware catalogs are i18n-4, deferred.
- Wildcard `*` matches `Locale::default()`.
- Drops `q=0` entries (RFC 7231: "not acceptable").
- Malformed q values are treated as 1.0 (lenient over
  strict because a strict reject leaves the user with the
  default rendering anyway, and lenient gives them a
  chance to land on the right locale).
- Ties resolve in document order so first-listed wins.

### `Locale::default() = Ja`

Existing v0.4 — v0.35.0 user-facing surfaces are
JA-hardcoded. The default-locale tie-breaker preserves
that for users without `Accept-Language` (mobile apps,
scripts, edge cases). Operators with majority-EN audiences
can switch the default later — the chosen Locale's "default"
status is one constant, not a deployed configuration knob,
because changing the default is a behavior change worth
making visible in the diff.

### Per-handler locale resolution

`cesauth_worker::i18n::resolve_locale(&Request) -> Locale`
is the single per-request locale lookup. Handlers call it
once at the top of the handler and thread the resolved
`Locale` through to the template. The resolver currently
just consults `Accept-Language`; future iterations may
layer on user-pref cookies and tenant defaults (cookie →
tenant → header → default).

The resolved locale is **stable for the duration of the
request**. Re-resolving per-template would risk a single
response page rendering with mixed locales (e.g., one
template observes the cookie, another does not because
the cookie was dropped in middleware).

### Backward-compatible migration

Each migrating call site adds a `_for(..., locale)` variant
alongside the existing function. The plain function becomes
a default-locale shorthand:

```rust
pub fn sessions_page(items, csrf, flash) -> String {
    sessions_page_for(items, csrf, flash, Locale::default())
}

pub fn sessions_page_for(items, csrf, flash, locale) -> String {
    // ... uses lookup(key, locale) for every visible string
}
```

This lets existing callers (and tests) continue to work
unchanged while new call sites adopt locale-awareness
incrementally. Once every call site is migrated the
shorthands can be removed in a future release; for now
they're not in the way.

## v0.36.0 migration scope

In-scope:

- **Flash banners** — `FlashKey::display_text` routes
  through `lookup`. Five keys: `TotpEnabled`,
  `TotpDisabled`, `TotpRecovered`, `LoggedOut`,
  `SessionRevoked`.
- **`/me/security/sessions` chrome** — page title, intro,
  empty-state message, back-link, current-device badge,
  current-device disabled button + title, revoke button,
  auth-method labels (passkey, magic_link, admin,
  unknown), session-meta labels (sign-in, last-seen,
  client, session-id).
- **TOTP enroll wrong-code error** — the
  "入力されたコードが一致しませんでした..." string.

Deferred (post-v0.36.0):

- Login page chrome (handle/password, "send magic link"
  button, etc.).
- TOTP enroll/verify/disable/recover page chrome.
- Magic Link request/verify pages.
- Security Center index page (the wrapping for the
  TOTP/sessions sections — sub-section content is i18n'd
  but the section headers and intro text are still
  hardcoded JA).
- Admin console (operator-facing, English by design;
  i18n there is i18n-3 territory if/when demand surfaces).
- Machine error bodies (out-of-scope for the user-facing
  track; OAuth error responses are spec-mandated English
  codes anyway).

## Open questions

- **Q1**: User-pref cookie. A `/me/preferences` page
  setting an explicit locale would override
  `Accept-Language` for this account. Useful when the
  browser's negotiation doesn't match the user's actual
  preference. Defer until there's an observable need.
- **Q2**: Tenant-default locale. `tenant_settings`
  could carry a default that overrides
  `Accept-Language` for this tenant's users. Tenant-
  scoped i18n is structurally separate from per-user;
  add later.
- **Q3**: Date/time format localization. The sessions
  page currently emits ISO-8601 UTC for created_at /
  last_seen_at. JA users may expect a different format.
  Deferred — needs both timezone awareness (cesauth has
  none yet) and locale-specific date format catalogs.
  Tracked under the i18n-4 region/script work.
- **Q4**: Pluralization. CLDR plural rules differ by
  language (e.g., zero/one/few/many/other). v0.36.0's
  catalog has zero strings that need pluralization; if
  one is added, the locale-aware integer-substitution
  facility comes with it.

## Considered alternatives (rejected)

- **`fluent-rs` or `gettext-rs` runtime catalogs**.
  Rejected: bring runtime parsing of catalog files (.ftl
  / .po / .mo) and the surface area that comes with
  that. cesauth has 22 keys at v0.36.0 — a closed Rust
  enum is the right tool for this size of catalog.
  Reconsider when the key count exceeds ~200, where the
  recompile-to-add-a-string friction starts to matter.
- **Macro-generated lookup**. A
  `define_messages! { TotpEnabled => { ja: "...", en:
  "..." } }` macro would shorten the catalog source.
  Rejected for v0.36.0: the long-form match is greppable,
  reviewable, and IDE-jumpable; a macro adds another
  layer to peel back when debugging. Reconsider if the
  catalog grows past 100 keys.
- **Locale type with region/script subtags**. v0.36.0
  ships only primary-subtag-aware (`ja`, `en`). Adding
  `ja-JP` / `ja-Jpan` / `en-US` / `en-GB` is the i18n-4
  scope; defer until real demand surfaces.

## Operator-visible changes

- Existing JA-default rendering is preserved for users
  without `Accept-Language` — no behavior change for the
  most common case.
- Users sending `Accept-Language: en` see the migrated
  surfaces in English.
- Mixed JA + English on a single page is impossible
  within a request (the resolver runs once per request,
  not per template).

## Tests

808 → 833 (+25). New coverage:

- 20 in `cesauth_core::i18n::tests` — locale parsing
  (case-insensitive, region-stripping, unknown returns
  None), `Accept-Language` parsing edge cases (empty,
  q-value priority, q=0 dropped, wildcard, malformed q
  treated as 1.0, all-unsupported falls through, tie-
  breaking in document order, browser-typical headers),
  catalog completeness (every key resolves in every
  locale to nonempty, no two keys share text within a
  locale).
- 5 in `cesauth_ui::templates::tests` — sessions_page
  English rendering (chrome, method labels, revoke
  button, current-device badge), default-locale
  shorthand still produces JA.
