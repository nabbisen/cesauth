# i18n policy

cesauth's user-facing UI is internationalised via a closed `MessageKey`
catalog (RFC: see ADR-013 — `docs/src/expert/adr/013-i18n.md`). This
page is the operator-side digest of the contracts that bind every
visible string and timestamp.

## Locale set

Closed two-locale set: `ja` and `en`. Adding a locale is a code change
to `cesauth_core::i18n::Locale` plus a compile-failure walkthrough of
the `lookup_*` group functions in `crates/core/src/i18n/mod.rs`.

Admin-console surfaces are JA-only (ADR-013). End-user surfaces
negotiate via `Accept-Language` and fall back to JA on no-match.

## Plural form (RFC 107, v0.73.0)

Plural agreement is handled by:

```rust
use cesauth_core::i18n::{lookup_plural, plural_for, MessageKey, Plural};

// What plural category does (locale, n) fall under?
let cat = plural_for(Locale::En, 5);
assert_eq!(cat, Plural::Other);

// Resolve a (key, locale, n) tuple to the correct plural variant.
let s = lookup_plural(MessageKey::SecurityRecoveryRemaining, Locale::En, 5);
// → "{n} valid recovery codes"  (caller substitutes "{n}" with "5")
```

Plural categories are a closed enum (`One`, `Other`). Locales that need
richer CLDR categories (Slavic / Arabic add `Few` / `Many` / `Two`)
require adding a variant — the existing `lookup_plural` match becomes
non-exhaustive and surfaces the missing arms at compile time. There is
no runtime CLDR-XML parsing and no `icu` dependency; the WASM size
budget is preserved.

A `MessageKey` is opt-in plural-aware: `is_plural_aware(key)` returns
the closed set. Calling `lookup_plural` on a non-registered key panics
loudly (the failure is a programming error, not a fallback case).

## Date / time rendering (RFC 111, v0.73.0)

**Policy: UTC ISO-8601 (RFC 3339) for every visible timestamp,
locale-independent.** The canonical formatter is
`cesauth_core::util::format_unix_as_iso8601(unix_secs: i64) -> String`,
which emits the `Z`-suffixed form (e.g., `2024-01-01T00:00:00Z`).

This choice is deliberate for two reasons:

1. **Ambiguity is unsafe.** Audit log entries, session list rows, and
   token expiration banners are security-sensitive surfaces. Timezone
   ambiguity ("was that 14:00 my time or 14:00 their time?") is a real
   incident-response failure mode. UTC ISO-8601 has exactly one
   interpretation; that's worth more than the ergonomic gain from
   local-time formatting.

2. **Operator-side tooling is already UTC.** Cloudflare Workers Logs,
   the R2 audit dump cron, and the `migrate export` CLI all emit UTC
   ISO-8601. Matching the UI lets operators paste a timestamp from a
   user report directly into a log search and find the corresponding
   row. The trace path costs zero conversions.

Per-user timezone preferences (`Asia/Tokyo` / `America/Los_Angeles` /
…) are separately useful and tracked as a future-work item; they
would supplement UTC ISO-8601 rather than replace it (think: "show
the local time but keep the UTC alongside for traceability").

### Caller contract

Any UI module that needs to render a timestamp MUST call
`cesauth_core::util::format_unix_as_iso8601` (re-exported as
`crate::util::format_unix_as_iso8601` from the core crate). Hand-rolled
formatters that use the `time` crate's `OffsetDateTime::from_unix_timestamp`
path are forbidden — the pin tests under
`crates/ui/src/admin/tests.rs::rfc_111` enforce this at the
canonical-formatter contract level.

### Numeric format

cesauth has no numeric-format locale rules today (no separators, no
currency). If a future surface needs them, the policy should match
the date policy: explicit, locale-independent, and minimal.
