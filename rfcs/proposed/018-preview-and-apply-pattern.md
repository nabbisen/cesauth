# RFC 018: Preview-and-apply pattern for destructive admin operations

**Status**: Proposed
**ROADMAP**: External UI/UX design update v0.50.1 — page 9 ("Operations UX: state → impact → execution → audit")
**ADR**: This RFC produces ADR-019 establishing the preview-and-apply convention; the pattern, once introduced, applies to several operations and warrants ADR-level documentation
**Severity**: **P2 — operator-experience pattern; not security-critical, but materially reduces operator-error class for destructive operations**
**Estimated scope**: Medium — ~200 LOC pattern infrastructure + per-operation adoption (small per operation); first adopters: RFC 017 audience editor + token rotation + safety controls

## Background

The UI/UX deck's page 9 frames operations UX as
"状態 → 影響 → 実行 → 監査" — state, then
impact, then execution, then audit. The current
admin console largely lacks the **impact** step:
edit forms typically go directly from "show
current value" → "submit new value", with the
operator's only intermediate signal being their
own mental model of what the change does.

Examples of where this gap shows up today:

- **`config_edit`** (`/admin/console/config`):
  changing `LOG_LEVEL` from `info` to `debug`
  applies on next request. The form shows the
  new value and a save button. The operator gets
  no preview of what's about to change, no
  rollback hint.
- **Admin token rotation**: same pattern. Click,
  confirm, applied.
- **Safety control toggles**: same pattern.
- **(Future) RFC 017 audience scoping editor**:
  same pattern unless we change it.

For each, the audit log records what happened —
but the operator's only opportunity to catch a
mistake is *before* clicking, with whatever
mental model they brought to the screen.

The deck's page 9 mock illustrates a different
shape:

```
Config preview / apply

  Current env values
  Pending change
  Impact: auth flow / audit
  Rollback hint
  [ Apply with audit ]
```

A two-step form: edit produces a preview;
preview shows "what's about to change"
explicitly; apply commits. The pattern is
common in infrastructure tooling (Terraform's
`plan` → `apply`, Wrangler's `--dry-run`) but
not in cesauth's admin console.

This RFC introduces the pattern for cesauth and
applies it to the smallest set of operations
where it pays off most.

## Requirements

1. Cesauth MUST provide a reusable
   "preview-and-apply" infrastructure: a route
   handler pattern, a template helper, and an
   audit convention.
2. Destructive admin operations (changes that
   affect auth flow, audit visibility, or other
   tenant-impacting state) MUST adopt the
   pattern.
3. The preview MUST show: current value, pending
   value, **operator-readable impact summary**,
   and **rollback hint** (how to reverse the
   change).
4. The apply step MUST be a separate POST
   distinct from the preview's GET, with CSRF
   protection on both.
5. The audit event emitted at apply MUST
   reference the preview event (a preview-applied
   pair correlation).
6. Operators with view-only roles MUST be able
   to reach the preview step but not the apply
   step.

## Design

### The pattern

For each destructive operation:

```
GET  /admin/.../config          ← view (existing)
POST /admin/.../config/preview  ← compute the diff, render preview page
POST /admin/.../config/apply    ← commit the change
```

The preview is **computed**, not just shown
back. It carries:

- The **before** state (canonical, fetched fresh).
- The **after** state (operator's input).
- The **diff** as a structured value the
  template can render row-by-row.
- An **impact statement** — operator-readable
  prose generated by the operation-specific
  service.
- A **rollback hint** — operator-readable prose
  describing how to undo. Not necessarily a
  one-click button (some changes can't be
  one-click-reversed); always at minimum a
  description.
- A **preview token** — an HMAC-signed payload
  carrying the `(operation_id, before, after,
  preview_ts, csrf)` tuple, signed with the
  session HMAC key. The apply step verifies the
  token, ensuring "the apply matches the
  preview the operator saw".

### The preview token

```rust
struct PreviewToken {
    operation_id: String,           // e.g. "config.log_level"
    before:       serde_json::Value,
    after:        serde_json::Value,
    preview_ts:   i64,
    csrf:         String,
    sig:          [u8; 32],          // HMAC-SHA256 over the rest
}
```

Wire form: `b64url(json) + '.' + b64url(sig)`,
embedded in a hidden form field on the apply
form.

The token's role: **bind the apply request to
the specific preview the operator saw**. An
attacker cannot tamper with the after-value or
craft an apply directly — the apply server
verifies the HMAC and confirms the after-value
matches what was previewed.

The token expires after 5 minutes
(`preview_ts` check at apply time). This
prevents stale preview-then-apply across
sessions.

### Impact statement contract

The operation's service computes the impact
statement. Each operation defines a small pure
function:

```rust
pub fn config_log_level_impact(
    before: &str,
    after: &str,
) -> ImpactStatement {
    ImpactStatement {
        title:    "Logging verbosity change",
        bullets:  vec![
            format!("Records at level '{after}' and above will be emitted."),
            // ...
        ],
        rollback: format!("Set LOG_LEVEL back to '{before}' and re-deploy."),
        severity: ImpactSeverity::Low,
    }
}
```

`ImpactSeverity::{Low, Medium, High}` — the
template colors the preview accordingly. High
severity (e.g. token rotation, revoke-all) gets
a red banner; medium gets a warning banner; low
gets info.

The operation-specific impact functions live in
`cesauth_core::admin::impact::*` — pure, easily
tested, reviewable.

### Adoption — first three operations

This RFC scopes adoption for three specific
operations. Other destructive admin operations
adopt the pattern incrementally as they're
touched.

#### A. `LOG_LEVEL` change (medium severity)

Existing: `/admin/console/config_edit` directly
applies. After: GET form → POST preview → POST
apply.

Impact statement: "Records at level X and above
will be emitted. Records below this level will
be dropped silently."

Rollback: "Set `LOG_LEVEL` back to the previous
value and re-deploy."

#### B. RFC 017's audience scoping change (medium severity)

The newly-introduced audience editor (RFC 017)
adopts preview-and-apply from PR 1. Impact
statement explains:

- "Client X may only introspect tokens with
  audience '<after>' from now on."
- "Tokens already issued continue to verify;
  introspection of those tokens by client X will
  start returning {\"active\": false} for those
  whose aud doesn't match."
- "Audit events `IntrospectionAudienceMismatch`
  will fire for cross-audience attempts."

Rollback: "Change the audience back to
'<before>'. Tokens that were rejected during the
mismatch window may be re-introspected
successfully."

#### C. Admin token rotation (high severity)

Existing: `/admin/console/tokens` direct rotate.
After: preview shows the rotation impact
explicitly:

- "Token <id> will be invalidated. Operators
  using this token in their tooling must update
  their config."
- "The replacement token is shown ONCE on the
  next page. cesauth cannot recover it."

Rollback: "Cannot be undone. The previous token
is irretrievable. Generate a new token via the
same flow if access is needed."

**High severity** rendering: red banner at the
top of preview ("DESTRUCTIVE — cannot be
undone"). Apply button text changes from "Apply"
to "Rotate token" — matching what the operator
is about to actually do.

### Audit cross-link

Two audit events per operation:

- `OperationPreviewed` — emitted at the preview
  step. Payload includes the operation_id,
  before, after, requesting actor.
- `OperationApplied` — emitted at the apply
  step. Payload includes the operation_id,
  before, after, applied_at, **preview_ts** (so
  audit reader can verify operator saw the
  preview before applying).

Both events carry the `request_id` from RFC 015
so a preview-apply pair share a request-level
correlation.

`OperationPreviewed` events without a matching
`OperationApplied` are an interesting audit
signal: they indicate operator considered a
change but didn't commit. Useful for incident
forensics ("did the operator nearly apply
this?") and for usage analytics.

### Routes adopted by this RFC

| Operation | Old route | New routes |
|---|---|---|
| LOG_LEVEL change | `POST /admin/console/config_edit` | `POST /admin/console/config_edit/preview`, `POST /admin/console/config_edit/apply` |
| Audience scoping | (RFC 017 PR 2 directly adopts) | `POST /admin/t/<slug>/oidc-clients/<id>/audience/preview`, `.../apply` |
| Token rotation | `POST /admin/console/tokens/rotate` | `POST /admin/console/tokens/rotate/preview`, `POST /admin/console/tokens/rotate/apply` |

Old routes return 308 to the new preview route
for one minor release, then are removed in the
following release.

### Read-only admin behavior

A read-only admin reaching the preview step:

- Can submit the form (computes preview).
- Sees the rendered preview WITH a banner
  noting "You don't have permission to apply
  this change."
- Apply button disabled / not rendered.

The preview itself is non-destructive (it's a
computed view of "what would change"); allowing
read-only admins to see the preview supports
"could a tenant admin do X here?" investigation
without granting write permission. The disabled
apply enforces the boundary.

## Test plan

### Pure (impact functions)

Per operation, ~3-5 unit tests covering
boundary cases:

1. Each `*_impact` function returns severity
   matching the documented level.
2. `rollback` text references the before-value
   when reversibility is value-dependent.
3. `bullets` are non-empty.

### Preview token

4. `preview_token_round_trip` — pin
   serialize/verify identity.
5. `preview_token_tampered_after_value_rejected`
   — pin HMAC verification.
6. `preview_token_expired_after_5_minutes_rejected`
   — pin TTL.
7. `preview_token_with_wrong_csrf_rejected` —
   pin csrf-in-token defense.

### Handler

Per operation:

8. `preview_renders_with_diff_and_impact`
9. `preview_with_no_change_shows_noop_message`
   — operator submits without modifying; preview
   says "No change. Apply will be a no-op."
10. `apply_with_valid_preview_token_commits`
11. `apply_with_tampered_preview_token_rejects`
12. `apply_emits_paired_audit_events`
    (`OperationPreviewed` + `OperationApplied`
    with shared request_id).
13. `apply_after_token_expiry_rejects_with_error`
14. `read_only_admin_can_preview_but_cannot_apply`

### UI rendering

15. `preview_high_severity_renders_red_banner`
16. `preview_medium_severity_renders_warning_banner`
17. `preview_low_severity_renders_info_banner`
18. `preview_includes_rollback_hint`

## Security considerations

**HMAC key reuse**. The preview token is signed
with the session HMAC key. The same key signs
session cookies. Reuse is acceptable: the
preview token is short-lived (5 min) and
binds (operation_id, before, after, csrf,
preview_ts) — even if leaked, it's only
useful as a replay against the same session
within 5 minutes for the same change. If a
future security review wants stricter
isolation, a derived key
(`HMAC(session_key, "cesauth preview token v1")`)
follows the v0.31.0 flash cookie pattern and
costs nothing to introduce.

**Token forgery**. HMAC-SHA256 with a 256-bit
key over a small payload is the standard JWS
shape. Same threat model as session cookies.
Test 5 pins tamper detection.

**Replay across sessions**. The token's CSRF
field is the session-bound CSRF cookie value.
A token captured from session A cannot apply
in session B because session B's CSRF cookie
differs. Pin via test 7.

**Privilege escalation through preview**.
Read-only admins can compute previews. The
preview itself reveals the **current state**
(operator already had read on this surface) and
the **proposed after state** (the operator
typed it themselves). No new disclosure. The
disabled apply enforces the actual privilege
boundary.

**Apply-time recheck**. The apply step **must
re-verify authorization**. A user who was a
Tenant Admin at preview time and a Read-Only
Admin at apply time gets denied at apply. The
preview token does not carry authorization —
it carries the operation, not the operator's
permission. The apply handler runs the same
permission check as a fresh request.

**Apply-time re-fetch**. The apply step
**should also re-fetch the current value** and
verify it matches the preview's `before`. If
another admin changed the same value between
preview and apply, the apply rejects with
"State changed since preview. Re-preview to
see current state." This prevents lost-update
problems in concurrent admin scenarios.

**Audit chain integrity**. The two audit
events (`OperationPreviewed`,
`OperationApplied`) integrate with the
existing hash-chained audit log (ADR-010). No
chain-format change. Preview-without-apply is
a single hash-chained row; preview-with-apply
is two rows in chain order.

## Open questions

**Should the preview show a "validate without
applying" diff for ANY operation, even
non-destructive ones?** No. Non-destructive
operations (read-only views, list-fetches) do
not benefit. The pattern's value is in the
**explicit consideration step before
mutation**. Spreading it to every form would
dilute the signal.

**Should there be an automatic apply timeout?**
i.e., if the operator previews and walks away
for 4 minutes 50 seconds, does the apply
button disappear? The 5-minute token TTL
already enforces a hard ceiling. A countdown
visible on the preview page is a UI nicety
but not RFC-required.

**Should we apply the pattern to per-row
session revoke?** No. Per-row revoke is
already a one-click destructive action with a
clear understanding by the operator (the
session is named, the impact is "this device
loses access"). Adding a preview step adds
ceremony without preventing real errors. The
**bulk** revoke-all-other-sessions surface
similarly: it's already two clicks (button +
confirmation page) which serves the same
purpose with less infrastructure.

The pattern targets operations where the
operator's mental model and the actual change
can plausibly diverge. Single-target session
revoke isn't that class.

## Implementation order

1. **PR 1** — Pure infrastructure: `ImpactStatement`,
   `ImpactSeverity`, `PreviewToken` in
   `cesauth_core::admin::preview`. ~120 LOC
   + 7 tests (items 4-7 + 1-3 for one example
   operation).
2. **PR 2** — Template helpers:
   `preview_page_for(diff, impact, locale)` in
   `cesauth_ui::admin::preview`. ~80 LOC + 4
   rendering tests (items 15-18).
3. **PR 3** — Adopt for `LOG_LEVEL` change.
   Routes, handlers, MessageKey variants. ~100
   LOC + 5 tests.
4. **PR 4** — Adopt for token rotation.
   ~80 LOC + 5 tests.
5. **PR 5** — Adopt for audience editor (RFC
   017 PR 2 alternative). ~50 LOC. RFC 017's
   PR 2 is rewritten on this base if RFC 018
   ships first; otherwise RFC 017 ships
   non-preview and refactors when RFC 018
   lands.
6. **PR 6** — ADR-019 graduates to Accepted.
   CHANGELOG. Operator chapter
   `docs/src/deployment/preview-and-apply.md`
   describing the pattern from operator
   perspective.

## Notes for the implementer

- **PR 1 is the linchpin.** All adopters
  depend on it. Land it first.
- The pattern naturally extends to future
  operations. Document in
  `docs/src/expert/development.md` (the dev
  directive): "When adding a destructive admin
  operation, adopt preview-and-apply unless
  the operation is unambiguous and
  single-target."
- Coordinate with RFC 017: if 017 ships first
  (without preview-and-apply), 017's audience
  editor refactors when 018 lands. The
  refactor is small — replacing one POST with
  two and threading the preview token.
- **Preview token signing key**: reuse session
  HMAC key for v0.50.x. If future review
  requires derived-key isolation, follow the
  v0.31.0 flash cookie HKDF pattern.
- The `OperationPreviewed`-without-
  `OperationApplied` audit signal is a real
  forensic asset. Tell operators about it in
  the deployment chapter so they recognize it
  during incident review.
